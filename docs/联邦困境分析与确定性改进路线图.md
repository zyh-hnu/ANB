1. **P0：打破 `ACC-ASR` 零和博弈**（第一优先级）
2. **P1：稳定突破 FreqFed 防御**（第二优先级）

---

## 1. 重新定义问题（避免“调参陷阱”）

你当前困境不是单一超参数问题，而是一个耦合系统问题：

- 攻击强度（`poison_rate`, `scaling_factor`）决定后门可学习性；
- 联邦异质性（Dirichlet Non-IID + 多本地步）放大客户端漂移；
- FreqFed 按频域特征聚类，会惩罚“恶意客户端同质更新”。

因此，单纯横向扫参会出现固定现象：

- 强攻击：`ASR` 高但 `ACC` 崩；
- 弱攻击：`ACC` 回升但 `ASR` 塌；
- 即出现 `ACC-ASR` 零和。

---

## 2. 根因链路（面向攻击者视角）

### 2.1 为什么 `ACC-ASR` 会零和

在当前实现中，恶意目标和主任务目标高度冲突。可抽象为：

- 主任务损失：$L_{clean}$
- 后门损失：$L_{bd}$
- 防御规避相关损失：$L_{stealth}$

若仅增大 $L_{bd}$ 权重，模型会牺牲判别边界导致 `ACC` 降低；  
若仅压制扰动保证 `ACC`，后门信号又会在 FedAvg 下被稀释，`ASR` 降低。

### 2.2 为什么容易被 FreqFed 盯上

FreqFed 本质检测的是“客户端更新在频域上的群体相似性和异常性”。  
如果恶意客户端共享过于一致的频谱模式，会在聚类中形成独立簇，被筛除或降权。

结论：必须同时做两件事：

1. 让恶意更新保留主任务能力（破零和）；
2. 让恶意更新在频域上“分散但协同”（规避 FreqFed 聚类）。

---

## 3. 新攻击策略：双目标解耦 + 频域反聚类

建议把现有 SAFB 升级为一套**两层优化攻击策略**：

### 3.1 层A：破零和（任务解耦）

恶意客户端局部训练损失改为：

$L_{mal}=L_{clean}+\lambda_{bd}L_{bd}+\lambda_{cross}L_{cross}+\lambda_{prox}\|w-w_g\|_2^2+\lambda_{anchor}L_{anchor}$

其中：

- $L_{clean}$：恶意客户端仍保留清洁样本学习；
- $L_{bd}$：触发样本到目标类；
- $L_{cross}$：触发但不改标签（稳定 `ACC`）；
- $\|w-w_g\|_2^2$：限制恶意更新偏移过大（抑制 ACC 崩盘）；
- $L_{anchor}$：与上一轮全局模型蒸馏/一致性约束。

### 3.2 层B：破聚类（针对 FreqFed）

在恶意客户端之间引入“分散协同”：

1. **频域分片轮换**：每轮切换/扰动 shard，不固定单一低频中心；
2. **相位与幅度去同质化**：不同恶意客户端施加不同相位族与幅度上限；
3. **更新方向去相关**：控制恶意客户端更新余弦相似度上限；
4. **分工攻击**：恶意客户端分为
   - `carrier`（主任务保持）
   - `injector`（后门增强）
   最终聚合后同时保 `ACC` 与 `ASR`。

---

## 4. 训练日程（必须分阶段）

为打破零和，必须避免“全程强攻”。建议 3 阶段：

### 阶段 I：Warmup（0~20% rounds）

- `poison_rate` 低（如 `0.2~0.4`）
- `scaling_factor` 低（如 `1.5~2.5`）
- 目标：先稳定 `ACC` 表征底座。

### 阶段 II：Co-Train（20%~80% rounds）

- 逐步上调 `poison_rate` 和 `scaling_factor`
- 启用 `cross` 与 `anchor/prox` 约束
- 目标：在不击穿 `ACC` 的前提下抬升 `ASR`。

### 阶段 III：Consolidate（80%~100% rounds）

- 固定或轻回落强度，做“后门巩固 + 频域分散”
- 目标：稳定最终 `ASR` 且防止最后几轮 `ACC` 回落。

## 5. 针对当前代码的落地改造点

### 5.1 `federated/client.py`（优先改）

新增或强化：

1. `cross_ratio` 数据路径（触发不改标签）；
2. `prox_mu`（FedProx 风格本地约束）；
3. `anchor_weight`（logit/feature 蒸馏约束）；
4. `malicious_role`（`carrier`/`injector`）。

### 5.2 `core/attacks.py`

新增：

1. shard 轮换调度器（随轮次与客户端变化）；
2. 相位池分组与幅度上限约束；
3. “频域相似度惩罚接口”供训练时调用。

### 5.3 `federated/server.py`

新增监控：

1. 恶意客户端间更新余弦相似度；
2. 恶意与良性更新范数比；
3. 每轮被防御拒绝比例与最终 bypass。

## 6. 实验矩阵（按顺序跑，避免盲扫）

### Step A：只验证“破零和”（先关防御）

- `defense_enabled=False`
- 跑 3 组：
  - A1：现有基线
  - A2：+ `cross_ratio`
  - A3：+ `cross_ratio + prox + anchor`

通过标准：`ACC` 显著提升且 `ASR` 不低于基线 85% 水平线。

### Step B：加入 FreqFed，验证“破聚类”

- `defense_method=hdbscan`
- 跑 3 组：
  - B1：A3 + 固定 shard
  - B2：A3 + shard 轮换
  - B3：A3 + shard 轮换 + 去同质化约束

通过标准：`Bypass` 与 `ASR` 同时提升，`ACC` 不显著下降。

### Step C：最终门槛复验（3 seeds）

种子：`42, 123, 2026`，轮次：`30` 和 `50`。  
最终验收门槛：

- `ACC >= 60%`
- `ASR(single) >= 85%`
- `ASR(multi) >= 80%`
- `Bypass >= 70%`

## 7. 你要的“确定性”如何定义

这里的“确定性”定义为：**可复现地满足验收门槛**，不是理论上的全局最优保证。  
满足以下条件即判定“困境已被解决”：

1. 3 seeds 全部过门槛；
2. 30/50 rounds 都过门槛；
3. 不是依赖异常 fallback（例如防御报错后全放行）；
4. `history.json + 配置快照 + 关键日志` 完整可追溯。
